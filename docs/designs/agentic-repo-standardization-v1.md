## Problem Statement
The repo has strong technical assets (backend, frontend, CLI, MCP, and deep docs), but a new AI agent cannot reliably cold-start without manually scanning many markdown files and code paths. This creates onboarding friction, token waste, and inconsistent execution paths between sessions.

The desired outcome is an agentic operating model where a fresh agent quickly understands architecture, workflow, constraints, and where to act first.

## Demand Evidence
Current demand evidence is weak and pre-product:
- No external user or paying signal exists yet.
- The founder is the immediate user and reported repeated startup-session overhead.
- Concrete demand proxy from session: when starting a new agent, context recovery is slow and requires broad repo exploration.

Interpretation: this is a high-leverage internal tooling bet for founder velocity, not yet market-validated product demand.

## Status Quo
Current workflow:
- Founder manually creates a plan and TODOs.
- Agent executes against that plan.
- Founder performs rigorous manual plus automated testing.

Current costs/frictions:
- Entire product was built in 2-3 days, indicating high execution speed but limited process hardening.
- No single onboarding contract file for agents (`CLAUDE.md` / root `AGENTS.md` absent).
- Docs are rich but fragmented across `docs/agent-docs`, `docs/steps`, root spec files, and component readmes.

## Target User & Narrowest Wedge
Primary user now:
- Founder/operator (developer + CEO) running multi-session agentic development.

Narrowest wedge worth shipping now:
- A canonical repo agent contract + lightweight doc taxonomy that enables any new agent to start correctly from one entrypoint and route to scoped docs with minimal token usage.

## Constraints
- Hackathon-origin codebase transitioning toward startup-grade process.
- Must preserve velocity; avoid heavyweight governance.
- Must support multiple agent ecosystems (Codex, Claude, Copilot).
- Existing docs contain useful context and should be reorganized/refactored, not discarded.

## Premises
1. The highest-value bottleneck is agent cold-start quality, not feature throughput.
2. One giant instruction file is brittle; layered docs with scoped precedence is more robust.
3. The repo needs a single canonical source of truth with thin compatibility adapters for different agent tools.
4. Documentation must be operationally maintained with freshness checks, or it will decay quickly.

## Approaches Considered
### Approach A: Agent Bootstrap MVP
Summary:
- Add one canonical guide in root and thin compatibility pointers.
- Minimal structure and no governance automation.

Effort: S
Risk: Low
Pros:
- Fastest immediate onboarding improvement.
- Minimal disruption.
Cons:
- High risk of drift over time.
- Limited scalability for multi-agent workflows.
Reuses:
- Existing root and docs markdown files.

### Approach B: Agentic Operating System v1
Summary:
- Introduce layered instruction architecture, docs taxonomy, ownership, and drift checks.
- Optimize for durable multi-agent execution quality.

Effort: M
Risk: Medium
Pros:
- Strong long-term reliability.
- Lower ambiguity and token burn.
- Scales from solo founder to team.
Cons:
- More initial setup work.
- Requires documentation discipline.
Reuses:
- Current docs and runbooks as source material.

## Recommended Approach
Approach B is recommended because the objective is startup-grade refinement, not temporary patching. The design should maximize repeatability across sessions, agents, and future contributors.

### Proposed Standardization Blueprint
1. Canonical agent entrypoint:
- Add root `AGENTS.md` as the canonical operational contract.
- Include: architecture map, command matrix, test matrix, repo navigation, decision rules, and "first 10 minutes" bootstrap.

2. Compatibility surfaces:
- Add root `CLAUDE.md` that references/aligns with canonical guidance.
- Add `.github/copilot-instructions.md` as a thin compatibility adapter.
- Keep canonical content centralized to avoid triple-maintenance.

3. Scoped instruction layers:
- Add sub-tree `AGENTS.md` files for:
  - `documind/backend/`
  - `documind/frontend/`
  - `documind/documentation/`
  - `demo-test-app/` (if actively used)
- Scope each file to local build/test rules and invariants.

4. Documentation taxonomy normalization:
- `docs/architecture/` for system shape and invariants.
- `docs/decisions/` for ADR-style records.
- `docs/runbooks/` for operational flows (setup, tests, release, debug).
- `docs/agent/` for agent-facing policies and workflow overlays.
- Move legacy files incrementally with alias/index pointers to preserve discoverability.

5. Operating rhythm files:
- Add root `TODOS.md` (live sprint focus, no historical log bloat).
- Add root `DECISIONS.md` (short tactical decision ledger).
- Keep `SPEC.md` as product thesis, not daily execution log.

6. Validation and drift controls:
- Add a lightweight `docs-index.md` inventory with owner + last-updated + source-of-truth tag.
- Add CI check that fails when required root contracts are missing (`AGENTS.md`, `CLAUDE.md`, `TODOS.md`) or stale timestamps exceed threshold for critical docs.
- Add onboarding smoke test: "fresh agent can produce accurate architecture summary in <=10 min."

7. Agent execution protocol:
- Define standardized lifecycle for all agent tasks:
  - Discover -> Plan -> Execute -> Verify -> Summarize -> Update `TODOS.md`.
- Include strict file-reference and verification expectations.

## Open Questions
1. Should `SPEC.md` be split into product spec + technical architecture doc, or remain monolithic?
2. What freshness SLA is acceptable for critical docs (e.g., 14 days vs 30 days)?
3. Should `demo-test-app` remain first-class or be demoted to optional experimental scope?

## Success Criteria
1. A new agent can explain repo architecture and active priorities after reading canonical entry docs in under 10 minutes.
2. Agent-generated plans consistently route work to correct subsystem without manual correction.
3. Cold-start token usage and setup time materially decrease versus current baseline.
4. Foundational docs stay synchronized with code changes through lightweight drift checks.

## Distribution Plan
This deliverable is a repo-native operating standard distributed through git.
- Channel: repository root + docs tree.
- Consumers: AI coding agents and human collaborators.
- CI/CD: include doc-contract checks in existing PR verification pipeline.
- Rollout: merge as a single standards PR, then require all future feature PRs to follow the new doc contract.

## Dependencies
- Founder time to normalize/relocate existing docs.
- Agreement on canonical file ownership and update cadence.
- CI pipeline hook for doc checks.

## The Assignment
Run a blind onboarding test with 3 fresh agent sessions:
- Give each session only the canonical entry docs.
- Ask for: architecture summary, first safe task, and execution plan.
- Record where they fail or hallucinate, then patch the standard until all 3 sessions converge on the same correct understanding.
